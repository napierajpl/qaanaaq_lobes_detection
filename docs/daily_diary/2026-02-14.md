# Daily Diary - February 14, 2026

## Overview

This entry documents the **extended training set** (background tiles + augmentation), why we use it, and the new **train subsampling** option (30% per epoch) to shorten epoch time and vary the training set each epoch. E2E conftest was updated so minimal tests do not require the extended set; changes pushed.

---

## Why we use background tiles

The raw training split is dominated by **lobe** tiles (tiles that contain solifluction lobes). If the model only sees lobe examples, it tends to over-predict lobes on background (non-lobe) areas. **Background tiles** are tiles that do not contain lobes. We add them to the training set so the model learns “no lobe here” as well as “lobe here,” improving class balance and reducing false positives. The extended set is built by `prepare_extended_training_set.py`; the number of background tiles is controlled by a ratio to (lobe + augmented lobe) in `data_preparation_config.yaml` (e.g. 1:1). Candidates with too many white pixels are excluded via `white_threshold`.

---

## Why we use augmentation

We have a limited set of **lobe** tiles. **Augmentation** increases effective diversity and helps the model generalize: same content under different rotations and lighting. We use **pre-written** augmentation (not on-the-fly at training time): the prep script writes rotated (0°, 90°, 180°, 270°) and contrast/saturation-varied versions of lobe tiles to disk and registers them in `extended_training_tiles.json`. Training then just loads these tiles; each epoch can sample from the full set (lobe + background + augmented). Augmentation is configured in `data_preparation_config.yaml` (rotations, contrast_range, saturation_range).

---

## Why we use subsampling

With the extended set, we have many tiles and thus many batches per epoch (e.g. 173). At ~2 minutes per mini-batch, one epoch takes hours and full training takes days. **Subsampling** means that each epoch we use only a **random fraction** of the training set (e.g. 30%). That gives roughly 30% of the batches per epoch, so epochs are much shorter and we get results sooner. Because the subset is **random and different every epoch**, we avoid overfitting to the same fixed subset and still expose the model to the full pool of tiles over many epochs.

---

## What was implemented

**Config (`configs/training_config.yaml`)**

- Under `data`: **`train_subsample_ratio: 0.3`** — each epoch uses a random 30% of training tiles. Use `1.0` to disable subsampling.
- Training already uses **`use_background_and_augmentation: true`**, so it loads `extended_training_tiles.json` (lobe + background + augmented) produced by `prepare_extended_training_set.py`.

**Training script (`scripts/train_model.py`)**

- Reads `train_subsample_ratio`; if &lt; 1.0, each epoch builds a new random subset of `train_tiles` with `np.random.default_rng(epoch)`, creates a dataloader from that subset, and runs one epoch. Full tile list is unchanged; only the loader’s subset changes per epoch.
- Logs `data.train_subsample_ratio` to MLflow when subsampling is used.

**E2E (`tests/e2e/conftest.py`)**

- `make_minimal_config` sets **`use_background_and_augmentation: False`** so the minimal training test uses only `filtered_tiles.json` and does not require `extended_training_tiles.json` (which is not created by the default dev data prep).

## Conclusion

Background tiles and augmentation are used to balance classes and increase lobe-tile diversity; subsampling at 30% shortens each epoch and varies the training set per epoch. Config key `train_subsample_ratio: 0.3`, per-epoch random subset in the training loop, and E2E conftest fix are in place; changes pushed.

## Later: SE + PPM, output bounding, dev-run fixes

### SE + Pyramid Pooling Module (PPM)

- **Implemented** backlog item 5: optional Squeeze-and-Excitation (SE) and Pyramid Pooling Module (PPM) on the deepest encoder feature (enc4) before the bottleneck in SatlasPretrain U-Net.
- **Config:** `model.use_se`, `model.use_ppm` (flat, default false). Modules are created only when enabled; ablation: baseline, +SE, +PPM, +both. Order when both on: PPM then SE.
- **Code:** `src/models/se_ppm.py` (SELayer, PyramidPoolingModule), integration in `SatlasPretrainUNet`; factory and `configs/training_config.yaml` wired. Not in Optuna search space.
- **Tests:** `tests/unit/test_se_ppm.py` (SELayer, PPM, SatlasPretrainUNet with/without SE/PPM). Backlog and `docs/project_features.md` updated.

### Output bounded to [0, 20]

- Predictions were sometimes negative; targets are proximity 0–20. **Output activation** added so model output stays in range.
- **Options:** `output_activation`: **clamp** (default, no saturation), **sigmoid**, **sigmoid_steep** (temperature). When `proximity_max: 20`, clamp gives full gradient in (0,20); sigmoid can saturate at extremes.
- **Config:** `model.proximity_max: 20`, `model.output_activation: clamp`; optional `sigmoid_temperature` for sigmoid_steep.
- **Code:** `_bound_proximity()` in `src/models/architectures.py`; UNet and SatlasPretrainUNet take `proximity_max`, `output_activation`, `sigmoid_temperature`. Factory passes from config.
- **Tests:** `tests/unit/test_output_activation.py` — _bound_proximity (clamp/sigmoid/sigmoid_steep), UNet output in [0,20], factory.

### Dev-run fixes

- **Missing dev/train_512:** With `tile_size: 512` and `--dev`, training expected `data/processed/tiles/dev/train_512/filtered_tiles.json`, which often doesn’t exist. **`--tile-size` override** added to `train_model.py`; use `--dev --tile-size 256` to use existing dev 256 data.
- **Missing extended set:** When `use_background_and_augmentation: true` but `extended_training_tiles.json` is missing (e.g. dev), training now **falls back** to the filtered_tiles split and logs a warning instead of raising. Normalization stats use all train tiles when extended wasn’t loaded.

**Illustrations:** After a training run, MLflow artifacts (e.g. `mlruns/<exp_id>/<run_id>/artifacts/plots/loss.png`, `mae_comparison.png`, prediction tiles) provide training curves and sample predictions.
