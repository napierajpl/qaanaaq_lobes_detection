## What We Added (Code)

### 1. SQLite storage for Optuna (default)

- **Default storage**: SQLite database at
  `data/optuna_studies/{study_name}_{mode}.db`
  e.g. `data/optuna_studies/lobe_detection_hp_tuning_production.db`
- **Resume**: On startup, the script calls `optuna.load_study()` when storage is set; if the study exists, it resumes (same trial numbering, sampler state). If not, it creates a new study.
- **CLI**:
  - `--storage URL` — custom Optuna storage URL (overrides default).
  - `--no-persist` — use in-memory study (not recommended for long or unstable runs).
- **Logging**: Prints `[STORAGE] database=...` and logs storage path so it’s clear where state is saved.

### 2. CSV export behavior with DB

- CSV at `data/optuna_results/lobe_detection_hp_tuning_trials.csv` is still written after each run.
- Export logic was updated to **merge** trials from the study (DB) with existing CSV rows: avoid duplicate trial_number per study_name, and update state/value/datetimes for trials already present (e.g. RUNNING → COMPLETE).
- Original session_id and session_started_at are preserved for existing rows; only new trials get the current session.

### 3. Run flow with persistence

- If the study already has enough completed trials (e.g. `n_trials=10` and 10 are done), the script skips `optimize()` and only exports to CSV.
- Otherwise it runs `optimize(n_trials=...)`; Optuna itself continues from the last trial id in the DB.
