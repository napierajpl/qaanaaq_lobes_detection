# .cursorrules

These rules define how the assistant should work inside this repository.
Code quality beats speed. If rules conflict or requirements are unclear, stop and ask.

---

## Language and runtime

- Use **Python 3.11+**.
- Prefer standard library first, then well-known packages.
- Write **short functions** with **meaningful names** and **one responsibility**.
- Follow **OOP and Clean Code** principles:
  - single responsibility
  - explicit dependencies
  - no hidden side effects
  - readable > clever

---

## Preferred tooling and stack

- Dependency management: **poetry**
- Create and use virtual environments
- Data processing:
  - prefer **pandas**
  - prefer **sql**
  - **do not use spark** unless explicitly requested
- Configuration:
  - strongly prefer **yaml** files as a user-facing interface
  - logic should read config, not hardcode behavior
- Formatting and hygiene:
  - assume **pre-commit** hooks are enabled
  - code must be compatible with common linters/formatters

---

## Development workflow

### Planning first (important)

- When the prompt is **complex or multi-step**:
  - write a **clear plan first**
  - identify unclear points
  - **ask questions before coding**
- Do **not rush into implementation**.
- Confirm assumptions early.

---

## Test-driven approach

- Use **test-driven development** by default.
- You may start by writing **unit tests first**, then implement.
- Prefer `pytest`.
- Tests should be:
  - deterministic
  - fast
  - focused on one behavior
- Refactor only after tests pass.

---

## When uncertain

- If something is unclear, **ask** instead of guessing.
- Explicitly ask whether:
  - a variable is always present
  - a field can be null/missing
- Avoid defensive code jungles:
  - many `if`s
  - wide `try/except`
- Validate at boundaries instead.

---

## Documentation and verification

- If unsure about behavior:
  - check **official documentation**
  - then search the internet if needed
- Never present guesses as facts.
- If something cannot be confirmed, say: **"I cannot verify this"**.

---

## Code style rules

### Imports

- Always put imports **at the top of the file**.
- If you believe imports must live elsewhere:
  - stop
  - ask explicitly before doing it

### Defaults and fallbacks

- Prefer **fast fail**: if a value or key does not exist, **do not** add default values silently.
- Prefer to **know exactly which values are used**: fail explicitly (e.g. `KeyError`, `ValueError`) when something is missing rather than substituting a default.
- Add fallbacks or defaults **only after confirmation with me**; do not assume sensible defaults.

### Control flow and repetition

- Avoid chained `if/elif` blocks.
  - consider loops, mappings, or polymorphism
- Avoid copy-paste logic.
  - extract helpers
  - reuse existing functions
- Prefer early returns and flat structure.

### Exceptions

- Catch only what you can handle meaningfully.
- Prefer explicit validation over silent failure.

### Typing

- Use type hints by default.
- Make public interfaces intentional and stable.
- use logging instead of prints

---

## File size and structure

- **Scrutinize files over ~200 lines**: consider splitting, extracting helpers, or **moving helpers into separate modules under `src/`** (e.g. `src/tuning/`, `src/data_processing/`) so the script stays thin orchestration only. Do this in the same refactor pass, not as a follow-up.
- Be alert when a file grows **beyond ~300 lines**.
- Prefer **factories or registries** over long `if/elif` chains when creating components (e.g. loss, model) from config.
- Extract **setup and logging** into small named functions so the main flow stays linear and readable.
- **Centralize** shared config/HP logic and domain helpers (e.g. proximity, paths) in `src/utils/` or a dedicated module; avoid duplicating the same logic across scripts.
- Ask whether it should be:
  - split into multiple files
  - split by responsibility
- Organize code as if this will grow into a **large project**:
  - `src/<package>/`
  - `infrastructure/`
  - `tests/`
  - `configs/`
  - `scripts/` (thin orchestration only)
- Keep business logic out of scripts and notebooks.

### Refactoring scripts (future passes)

- **Script list**: See `docs/refactoring_log.md` for script inventory and status.
- **Apply consistently**:
  - `get_project_root(Path(__file__))` (pass `Path(__file__)`, not `__file__`).
  - Open text files with `encoding="utf-8"` (config, CSV, JSON, filtered tile lists, etc.).
  - Use `path_key` from `data.tile_size` when config has both `paths[mode]` and `paths[mode]_512` (e.g. `path_key = mode if tile_size == 256 else f"{mode}_512"`).
- **Extract**: Repeated path/tile_dir logic into helpers (e.g. `_tile_dir(dev_mode, tile_size)`); CSV/config headers into named constants; “save results” into small functions.
- **Log decisions**: Note non-obvious choices or questions in `docs/refactoring_log.md`; resolve with user after the refactor pass.
- **Notes from first pass**: Scripts that don’t use project root or file I/O (e.g. `start_mlflow_ui.py`, `elapsed_time.py`) can stay Pending. For scripts >200 lines, consider a second pass to extract more helpers and flatten control flow; mark status in refactoring_log.

## Reuse before writing new code

- Check if similar functionality already exists.
- Prefer reuse or refactor over duplication.

## CI/CD, Docker, portability

- Assume execution in **GitLab CI/CD** and/or **Docker**.
- Never rely on:
  - absolute local paths
  - OS-specific behavior (Windows/macOS assumptions)
- Use:
  - relative paths
  - environment variables
  - config files
  - `pathlib`
- Assume **git bash** and **gitlab repositories**.

---

## Temporary scripts and experiments

- You may write **temporary scripts** to validate ideas.
- Place them in a `personal/` directory.
- Assume `personal/` is **gitignored**.
- Do not treat experimental scripts as production code.
- Explanation files and plans for implementation (generally all .md files that help me work and are not documentation) should be in personal folder.

---

## Solution process requirements

- When implementing:
  - consider **2–3 possible approaches**
  - choose the best based on:
    - simplicity
    - maintainability
    - testability
- After coding:
  - switch perspective
  - perform a **self code review**
  - check naming, cohesion, duplication, clarity, and tests

---

## Git safety

- Never commit or push without explicit user consent.
- If changes are wide-ranging, propose a plan first.

---

## Communication style

- Be concise and concrete. Cheerful just a little bit
- Clearly state:
  - assumptions
  - tradeoffs
  - open questions
- If assumptions were required, ask for confirmation before continuing.

## Project tracking and documentation

### Daily Diary

- **Location**: `docs/daily_diary/YYYY-MM-DD.md`
- **Purpose**: Track meaningful training runs, experiments, and insights
- **When to update**:
  - After significant model changes or experiments
  - When working on training, loss functions, or architecture
  - When analyzing MLflow runs that reveal important patterns
  - Focus on top ~20% of runs (most meaningful)
- **What to include**:
  - Key runs with configurations and results
  - What worked / what didn't work
  - Key insights and learnings
  - Metrics summaries
  - Documentation created
  - Lessons learned
- **What NOT to include**:
  - Future plans or "next steps" (go to improvements backlog instead)
  - Routine runs with no significant changes
- **Format**: See `docs/daily_diary/README.md` for guidelines

### Improvements Backlog

- **Location**: `docs/improvements_backlog.md`
- **Purpose**: Track potential improvements, ideas, and future work
- **When to update**:
  - When identifying potential solutions during analysis
  - When discussing future improvements
  - When ideas emerge from experiments
- **Organization**: By priority (High/Medium/Low) and category
- **Status tracking**: Mark items as completed when implemented

### Diary Maintenance

- **Check diaries** when working on models, training, or significant changes
- **Update relevant daily entries** if new insights emerge
- **Reference diary entries** in backlog items for context
- **Keep entries factual** - document what happened, not plans
